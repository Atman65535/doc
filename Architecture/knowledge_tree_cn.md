# 知识树 (Knowledge Tree, KTree)
本文将介绍“知识树”（KTree）的定义。
KTree 是一个用于指导迭代优化和进度规划的框架。它包含作为**可验收里程碑（Acceptable Milestones）的“节点（Nodes）”，以及作为通往里程碑的需求（Requirements）**的“边（Edges）”。
该树包含代表独立技术路径的分支，最终目标位于根节点。

## 构建知识树
KTree 与标准的数据结构树对齐。其中**节点（Node）代表一个可以达到的阶段性目标，而边（Edge）**代表通往该目标的路径，包括前置知识或软件环境准备。以下是一个分支的示例：
```text
       ...
        |
        | 边 (Edge):  
        | 学习读取 MPU6050 数据
        | 欧拉角表示法
        | 基础积分定义
        |
------------------------------
| 节点 (Node): 线性卡尔曼滤波  |
| 验收标准: 获取三轴姿态数据    |
------------------------------
        |
        | 边 (Edge): 
        | 学习四元数
        | 雅可比矩阵线性化近似 ....
        |
        ... 下一个节点 ...
```
>节点与边
>边是路径和需求，而节点是带有验收标准的具体目标。一旦达到验收标准，我们就应当果断通过该节点（不要在这里追求极致完美，但要在开始前仔细设计验收标准）。
>例如： 尽管卡尔曼滤波是一种方法，但“获取姿态”这一目标是可验收且可衡量的。我们通过不断达成“节点”来构建系统，而不是单纯堆砌“边”。

树的每个分支都应当像这样构成。由于每个分支都代表一条独立的技术路径，我们应尽力对所有技术进行解耦，以最大限度地减少不同子系统之间的相互影响。

最终，所有分支将汇聚到我们的终极目标。从卡尔曼滤波到姿态估计，再到火箭控制。上述示例仅展示了类似于深度优先搜索（DFS）的一条分支，完整的树应包含更深的节点和更多的分支。

## 知识树的使用
**最小可行性产品（MVP）**的核心哲学不是要求全栈精通，而是通过阶段性成果实现闭环迭代。

这一过程与知识树的推进是一致的。我们从最末端的叶子节点（或不含内容的初始节点）开始，逐步“点亮”节点。我们的第一个版本可能仅由几个稀疏的节点组成，而最终项目将包含所有节点。然而，即使第一个版本看起来很简陋，它也必须是可运行/可操作的。

>什么是 MVP？
>当我们开发某个东西时，通常不会一次性完成最终版本，而是不断进行迭代。第一个包含最稳定技术和基础系统、且能够跑通的版本被称为 MVP。随后的所有增强和改动都基于该版本进行。

## 修剪 (Pruning)
时刻关注进度。由于我们的核心哲学是 MVP，所有旁路分支（Bypath branches）应当被果断修剪，或者排在主干（Main trunk）之后。在构建完知识树后，仔细设计进度规划同样至关重要。
